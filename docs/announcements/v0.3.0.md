## LinkedIn — single post

**Title:** Shipped: PayBridge 0.3.0 — unified payment SDK for South African + global rails

If you've integrated payments in South Africa, you know the drill. Each provider has its own special snowflake — PayFast wants form-encoded MD5, Yoco wants Svix-style webhooks, Ozow needs SHA-512 over fixed-order fields. Stripe is its own world. Crypto rails (MoonPay, Yellow Card) are an entirely separate integration.

The result: every project has its own bespoke payment layer. Every project has the same bugs. Switching providers means rewriting auth, signing, webhooks, refunds.

PayBridge 0.3.0 collapses this into one TypeScript SDK:

→ One API across **8 fiat providers** (SoftyComp, Yoco, Ozow, PayFast, PayStack, Stripe, Peach Payments, Flutterwave) and **2 crypto providers** (MoonPay, Yellow Card).
→ A smart **router** that picks providers by cost / latency / priority and falls back automatically when one fails.
→ A **circuit breaker** so a busy provider doesn't poison your retry loop.
→ **Rate-limit aware** — 429 responses don't trip the circuit.
→ **Webhook idempotency** (new in 0.3.0) — pluggable in-memory or Redis store, deduplicates events by ID, throws `WebhookDuplicateError` on retries.
→ **Webhook signature verification** done correctly: timing-safe comparisons, replay windows where the provider supports them, documented limitations where they don't.
→ **Crypto on/off-ramp** via the same shape — fiat ↔ stablecoin without a second SDK.

Stack: TypeScript strict, native fetch + node:crypto only, **zero runtime dependencies**. 239 unit tests. CI on Node 18 / 20 / 22. MIT license.

## Hacker News — Show HN draft

**Title:** Show HN: PayBridge — unified TypeScript SDK for fiat and crypto payments

Hi HN. I shipped PayBridge 0.3.0 — a unified Node.js / TypeScript SDK that puts 8 fiat payment providers + crypto on/off-ramp behind a single API, with smart multi-provider routing, circuit breakers, and webhook idempotency.

Why it exists: integrating payments in South Africa (and most non-US markets) means re-implementing webhook signing, redirect flows, refund quirks, and currency edge cases for every provider. PayFast wants form-encoded MD5 with field ordering. Yoco uses Svix-style webhook envelopes. Ozow signs SHA-512 over a fixed concat. Stripe is its own world. Existing libraries cover one provider each, leaving you to glue them together.

Approach: one `PaymentProvider` abstract class, one `CreatePaymentParams` shape, one `WebhookEvent` discriminated union. Each provider's idiosyncrasies live in the driver. Multi-provider users get a `PayBridgeRouter` that picks providers by cost / latency / priority and falls back automatically.

Some details I'd appreciate critique on:
- The MoonPay widget URL signing was verified against a reference implementation I wrote separately, not by trusting the docs. It's the kind of code where the wrong leading-`?` makes all signatures invalid (and indeed an earlier release shipped that bug).
- 429-aware routing: rate-limit responses skip the provider for the request but don't count toward circuit-breaker failures. Reasoning: rate-limited ≠ broken; tripping the circuit would only push more load to the next provider.
- Webhook idempotency: opt-in store, dedupes by event ID with TTL. In-memory or Redis. Throws `WebhookDuplicateError` on retries so handlers can return 200 OK and shut up the provider's retry loop.
- Yellow Card crypto provider is gated behind `@experimental` because I couldn't find authoritative public docs for their partner API. Better to ship a clearly-broken driver with warnings than a confidently-wrong one.

Stack: TypeScript strict, native fetch + node:crypto only, zero runtime deps. 239 tests, CI on Node 18/20/22. MIT.

`npm install paybridge`
